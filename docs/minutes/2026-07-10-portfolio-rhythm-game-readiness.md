## 現在地

原型は「音ゲーとして遊べる最低線」には到達している。

- `menu.tscn` が入口。
- 4択メニューから、開発思想ADV、Claudeとのやり取りADV、ゲーム本体へ遷移できる。
- ゲーム本体は、縦画面720x1280、3レーン、BPM140、実曲同期。

- Godot の短時間起動確認は通った。

一方で、公開用ポートフォリオとしてはまだ調整が必要。

## 重要な差分

Drive議事録では、人の入力ログをAIが均して固定譜面にした、という流れが中心になっている。

しかし現在の `main.gd` は、`logs/chart_reconciled.json` や `chart_full.json` を正本として読んでいない。実際には、固定シードと生成規則で毎回同じ譜面を生成している。

これはゲームとしては成立するが、面接や提出時に「人が叩いた記録を譜面化した」と説明するなら、実装と説明がずれる。公開前に、次のどちらかへ揃える必要がある。

1. `chart_reconciled.json` をゲームが読むようにして、議事録どおりにする。
2. 議事録と説明を「人の設計則を生成規則に落とし、固定シードで凍結した」に修正する。

ポートフォリオの物語としては、1 のほうが強い。

## 外部比較から見た不足

FNF系の最低線は、入力が分かりやすく、音に合わせて押し、失敗条件が見えること。

Bemuse/osu!系の強みは、譜面ファイル、判定精度、スコア/精度/ランク、調整可能な表示、音との同期、公開ページから即プレイできること。

現在の原型は、音との同期と判定の中核はある。足りないのは、外へ見せる包装と、説明と実装の一致である。

## GitHub Pagesについての判断

ユーザーの認識はおおむね正しい。

スマホQRまで行くには、公開URLが必要になる。そのためには GitHub Pages か同等の静的ホスティングが必要である。

今回の作品は、本編 `witch-escape-and-kingdom` とは別の小さなポートフォリオなので、新しいリポジトリを作るのが自然。

推奨構成:

- リポジトリ名: `portfolio-battle-mini`
- 公開形式: GitHub Pages の project site
- Pages URL: `https://<ユーザー名>.github.io/portfolio-battle-mini/`
- Godot Web export の置き場: `docs/` または GitHub Actions の成果物

最初は分かりやすさ優先で、`docs/` 配下を Pages の公開元にするのがよい。
