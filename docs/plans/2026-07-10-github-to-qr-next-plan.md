## 推奨する公開構造

GitHub Pagesは、`main` ブランチの `/docs` フォルダを公開元にする方針が扱いやすい。

ただし、現在の `/docs` には議事録や計画書も入っている。GodotのWeb出力を `/docs` 直下に置くと、説明資料とゲーム本体が混ざりすぎる。

そのため、次の構造を推奨する。

- `/docs/index.html`
  - 公開トップページ
  - 作品名、簡単な説明、ゲーム開始リンクを置く
- `/docs/play/`
  - Godot Web版の本体
  - `index.html`、`.wasm`、`.pck` などを置く
- `/docs/plans/`
  - 計画書
- `/docs/minutes/`
  - 議事録
- `/docs/assets/`
  - QRコード画像や説明用画像

想定URLは次の形になる。

- トップページ: `https://sr22m24s9n-oss.github.io/portfolio-battle-mini/`
- ゲーム本体: `https://sr22m24s9n-oss.github.io/portfolio-battle-mini/play/`

QRコードは、最終的にはトップページURLに向けるのがよい。理由は、あとから説明文や注意書き、別版へのリンクを追加しても、QRコードを作り直さずに済むからである。

## 作業計画

### 1. Web exportの準備確認

Godot側でWeb書き出しが可能か確認する。

- GodotのWeb export templateが入っているか
- `export_presets.cfg` があるか
- なければWeb用プリセットを作る

ここで詰まる可能性があるのは、Web export templateが未導入の場合である。その場合は、Godot本体側でテンプレート導入が必要になる。

### 2. Web版を書き出す

GodotプロジェクトをWeb向けに書き出し、出力先を `/docs/play/` にする。

確認するものは以下。

- `docs/play/index.html`
- `docs/play/*.wasm`
- `docs/play/*.pck`
- 必要なJavaScriptやアイコン類

### 3. PCブラウザでローカル確認

Web版は、ファイルを直接開くだけでは正常に動かないことがある。そのため、ローカルの簡易サーバーで確認する。

確認項目:

- ページが開く
- 音が鳴る
- メニューから通常プレイへ進める
- 痕跡譜面プレイへ進める
- クリックまたはタップ入力が効く
- 画面サイズが大きく崩れない

### 4. 公開トップページを作る

`docs/index.html` を作る。

内容は最小限でよい。

- 作品名
- 短い説明
- 「ゲームを開く」ボタン
- 注意: 音が出る、スマホ縦画面想定、初回読み込みに少し時間がかかる可能性

ここでは派手なサイト化をしすぎない。目的は、QRから来た人が迷わずゲームを開ける入口を作ることである。

### 5. GitHubへ反映する

Web exportとトップページをコミットし、GitHubへpushする。

ここでGitHub上に、公開用ファイルが揃う。

### 6. GitHub Pagesを有効化する

GitHubリポジトリのSettingsからPagesを設定する。

推奨設定:

- Source: Deploy from a branch
- Branch: `main`
- Folder: `/docs`

設定後、公開URLが発行される。反映には数分かかることがある。

### 7. スマホ実機確認

公開URLをスマホで開く。

確認項目:

- トップページが開く
- ゲームが開く
- 音が鳴る
- タップ入力が効く
- 縦画面で破綻しない
- 痕跡譜面の保存と再プレイが動く

ここで問題が出た場合は、QRコード作成前に直す。

### 8. QRコードを作成する

スマホ確認が通った公開URLを使ってQRコード画像を作る。

保存先の候補:

- `docs/assets/portfolio-battle-mini-qr.png`

READMEまたは公開トップページにもQRコードを載せる。
