# 2026-07-10 計画書 - 譜面の正本を決める

## 目的

`portfolio-battle-mini` の譜面について、説明と実装が食い違わない状態を作る。

ここでいう譜面の正本とは、「どの時刻に、どのレーンへ、どの玉を落とすか」を決める唯一の参照先である。

## 現在の状態

現在の `main.gd` は、BPM140の曲に同期しながら、固定シードと生成規則で玉を出している。

主な規則:

- BPM: 140
- OFFSET: 0.08
- GAP_THRESHOLD: 1.1
- MAX_BURST: 4
- seed: 20260709

つまり、毎回同じ譜面にはなる。ただし、外部ファイルの譜面を読んでいるわけではない。

一方で、`logs/chart_reconciled.json` には、人の入力ログをAIが均した結果として、`notes` が保存されている。

この `notes` には、各玉の `id`、到達時刻 `t`、レーン `lane` がある。

## 問題

ポートフォリオの説明では、「人が叩く→AIが均す→固定する」という過程が強い。

しかし、ゲーム本体がその譜面ファイルを読んでいない場合、面接や提出時に説明責任が弱くなる。

このため、譜面の正本をどこに置くかを決める必要がある。

## 案A: `chart_reconciled.json` を正本にする

### 内容

`logs/chart_reconciled.json` の `notes` をゲームが読み、そこに書かれた時刻とレーンに従って玉を落とす。

### 良い点

- 「人が叩いた記録をAIが均した」という物語と実装が一致する。
- 譜面をファイルとして説明できる。
- 後から譜面を直す時、コードを触らずJSONを直せる。
- AI協働の成果物として見せやすい。

### 弱い点

- `logs/` は本来作業記録の場所なので、公開用正本を置く場所としては少し弱い。
- `chart_reconciled.json` は説明情報が多く、ゲームが読むファイルとしては重い。
- 現在の落下処理と、到達時刻指定の処理を合わせる必要がある。

### 改善した形

`logs/chart_reconciled.json` を直接読むのではなく、公開用に軽い譜面ファイルを作る。

候補:

- `charts/kyrgyz_techno_anthem.json`

中身の例:

```json
{
  "song": "kyrgyz_techno_anthem",
  "audio": "res://audio/kyrgyz_techno_anthem.ogg",
  "approach_time": 1.8,
  "notes": [
    { "id": 1, "t": 9.457, "lane": 1 },
    { "id": 2, "t": 11.606, "lane": 0 }
  ]
}
```

この場合、`logs/chart_reconciled.json` は「作業記録」、`charts/kyrgyz_techno_anthem.json` は「ゲームが読む譜面正本」と分けられる。

## 案B: 固定シード生成を正本にする

### 内容

今の `main.gd` の生成規則を正本として扱う。

譜面はJSONではなく、コード内の規則から生成される。

### 良い点

- 実装変更が少ない。
- 今のゲームの動きと一致している。
- `GAP_THRESHOLD` や `MAX_BURST` という「人の設計則をAIがルール化した」話は残せる。

### 弱い点

- 「人が叩いたログを均した」という説明と少しずれる。
- 譜面を目で確認しづらい。
- 面接時に「どこが譜面ですか」と聞かれた時、コード説明になりやすい。

### 向いている場合

短時間で公開URLを優先するなら、この案でも進められる。

ただし、説明文は「人の入力ログを直接採用した」ではなく、「人のフィードバックから生成規則を作り、固定シードで凍結した」に寄せる必要がある。

## 案C: 両方を残す

### 内容

ゲームは軽い譜面JSONを読む。

その譜面JSONを作る根拠として、`logs/chart_reconciled.json` と生成規則の考え方を説明資料に残す。

### 良い点

- 説明が強い。
- 実装も見せやすい。
- 作業記録と公開用データを分けられる。
- 後から譜面調整しやすい。

### 弱い点

- 案Aより少し作業が増える。
- 「作業ログ」「公開譜面」「実装」の3つを混同しない説明が必要。

## 推奨

推奨は案C。

理由は、ポートフォリオの主役が「完成したゲーム」だけでなく、「人の判断とAI協働の過程」だからである。

構造は以下がよい。

```text
logs/
  chart_reconciled.json        作業記録。人の入力をAIが均した証拠。

charts/
  kyrgyz_techno_anthem.json    ゲームが読む公開用譜面。軽く整えた正本。

main.gd                        譜面を読み、時刻に合わせて玉を落とす道具。

docs/
  minutes/                     経緯の議事録。
  plans/                       実装前の判断。
```

この分け方なら、`logs/` を人に見せる成果物の一つとして扱いながら、ゲーム本体は整理された `charts/` を読む。

## 実装方針

### 1. `charts/` を作る

`charts/kyrgyz_techno_anthem.json` を作る。

最初は `logs/chart_reconciled.json` の `notes` だけを抜き出し、ゲームに必要な最小情報へ整える。

### 2. `main.gd` に譜面読み込みを足す

起動時に譜面JSONを読む。

読む情報:

- audio
- approach_time
- notes

### 3. 玉の出現時刻を変える

現在は拍ごとに生成規則で玉を出している。

譜面JSON方式では、各ノーツの到達時刻 `t` から逆算して玉を出す。

例:

- 到達時刻: 9.457秒
- approach_time: 1.8秒
- 出現時刻: 7.657秒

玉は出現後、判定ラインへ向かって落ちる。

### 4. 落下速度を譜面と一致させる

今の `SPEED` 固定だと、出現から到達までの時間とずれる可能性がある。

安全な方法は、各玉に以下を持たせること。

- spawn_t
- hit_t
- lane

そして毎フレーム、

```text
進行率 = (現在時刻 - spawn_t) / (hit_t - spawn_t)
玉のY = spawn_y から base_y まで進行率で補間
```

こうすると、FPSや速度調整に左右されず、譜面の時刻が正本になる。

### 5. 判定は今の位置ベースを維持する

判定は今のままでよい。

理由:

- Web音声遅延の罠を踏みにくい。
- 見た目の玉と判定が一致する。
- スマホで理解しやすい。

ただし、将来の結果画面用にGOOD/BAD/MISSの数は記録できるようにしたい。

## 変更しないもの

この段階では、以下は触らない。

- 見た目の大改修
- ADVの背景や立ち絵
- ログ全体の整理
- スクリーンショット選定
- Web export
- QRコード作成

譜面の正本だけを先に固める。
