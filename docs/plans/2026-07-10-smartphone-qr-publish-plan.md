# 2026-07-10 計画書 - スマホQR公開までの道順

## 目的

`portfolio-battle-mini` を、スマホでQRコードから開いて遊べる状態にする。

この計画のゴールは、以下の3つ。

- GitHub上に公開用リポジトリを用意する。
- Godot Web版をGitHub Pagesで公開する。
- 公開URLをQRコード化し、スマホで実際に開けることを確認する。

## 現在確認できていること

- GitHub認証は有効。
- 新規リポジトリ `sr22m24s9n-oss/portfolio-battle-mini` は存在する。
- リポジトリはPublic。
- リポジトリはまだ空で、既定ブランチもまだ無い。
- ローカルの `C:\dev\portfolio-battle-mini` は、まだGitリポジトリではない。

## 基本方針

今回の作品は、本編 `witch-escape-and-kingdom` とは別の公開用ミニ作品として扱う。

そのため、`portfolio-battle-mini` を独立したGitHubリポジトリにする。GitHub Pagesは、この新規リポジトリから公開する。

最初は分かりやすさを優先し、Web公開物は `/docs` フォルダに置く方針にする。GitHub Pagesは、GitHub公式ドキュメント上でも、ブランチのルートまたは `/docs` フォルダを公開元として設定できる。

## 作業の順番

### 1. 公開前のローカル整理

目的: GitHubへ出す前に、不要な一時ファイルや重い作業ログを混ぜない。

作業:

- `.godot/` はGit管理しない。
- `logs/` は公開に必要なものだけ残すか、説明用に最小化する。
- `screenshots/` は必要な代表画像だけ残す。
- WAV音源はWeb公開には重いので、可能ならoggへ変換する。
- README / PROCESS.md / 議事録 / 計画書を整理する。

確認ポイント:

- 音源の権利・公開可否。
- ログを公開してよいか。
- ADV本文をそのまま外へ出してよいか。

### 2. 譜面の正本を決める

目的: 「人が叩く→AIが均す→固定」という説明と、実際のゲームを一致させる。

推奨:

- `logs/chart_reconciled.json` の `notes` をゲームが読む。
- 各ノーツの `t` と `lane` を使って、決まった時刻に決まったレーンへ落とす。

代替:

- 現在の固定シード生成を採用し、説明文を「人の設計則を生成規則へ落とし、固定シードで凍結した」に修正する。

判断:

- ポートフォリオの物語としては、前者のほうが強い。

### 3. Git管理を開始する

目的: ローカル原型をGitHubへ送れる状態にする。

作業:

- `C:\dev\portfolio-battle-mini` でGitを初期化する。
- 既存の `.gitignore` を確認する。
- 必要なファイルだけをステージする。
- 初回コミットを作る。
- GitHubリポジトリをremoteとして接続する。
- `main` ブランチをGitHubへpushする。

注意:

- GitHub側は空なので、READMEをGitHub画面で作らない方針は正解。
- 初回pushで `main` ブランチが作られる。

### 4. Web exportを作る

目的: Godotプロジェクトをブラウザで動く形にする。

作業:

- GodotのWeb exportテンプレートを確認する。
- `export_presets.cfg` を作る。
- Web出力を `/docs` に出す。
- 出力物の入口が `/docs/index.html` になるようにする。

確認ポイント:

- PCブラウザで開けるか。
- 音が鳴るか。
- タップ入力が効くか。
- スマホ縦画面で崩れないか。

### 5. GitHub Pagesを有効化する

目的: GitHub上の `/docs` をWebサイトとして公開する。

作業:

- GitHubの `Settings` を開く。
- `Pages` を開く。
- Sourceを `Deploy from a branch` にする。
- Branchを `main`、Folderを `/docs` にする。
- 保存する。

想定URL:

`https://sr22m24s9n-oss.github.io/portfolio-battle-mini/`

確認:

- GitHub Pagesは反映に少し時間がかかることがある。
- URLが404の場合は、数分待つか、Pages設定と出力ファイル名を確認する。

### 6. スマホ実機確認

目的: QRを作る前に、URLそのものがスマホで使えるか確かめる。

確認項目:

- ページが開く。
- 音が再生される。
- タップでレーン入力できる。
- 画面比率が大きく崩れない。
- ADVとゲームの往復ができる。

### 7. QRコード作成

目的: 面接・提出時に、スマホで即アクセスできる入口を作る。

作業:

- 公開URLからQRコード画像を作る。
- `docs/assets/` などに保存する。
- READMEや提出資料に貼れる形にする。

成果物:

- 公開URL
- QRコード画像
- README上のQR表示

## 完了条件

この作業は、次の状態になったら完了とする。

- GitHubリポジトリに `main` ブランチがある。
- GitHub Pages URLが発行されている。
