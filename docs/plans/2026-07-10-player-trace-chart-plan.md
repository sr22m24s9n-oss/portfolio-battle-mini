# 2026-07-10 計画書 - プレイヤーの痕跡を譜面に反映する最小実装

## 目的

通常プレイで人間のタップログを取り、その痕跡を次の譜面に反映する。

完成度の高い自動生成システムを作るのではなく、ポートフォリオとして「人間のフィードバックが記録され、AI協働の変換ルールを通り、次のプレイに道具として戻る」という思考を伝える。

## 前提

時間が限られているため、完璧な完成形は狙わない。

今回の到達目標は、以下の3点。

1. 通常プレイのタップログを保存する。
2. そのログから次回用の簡単な譜面を作る。
3. メニューに「痕跡譜面で遊ぶ」を追加する。

## この方針を採る理由

このゲームの主役は、音ゲー単体の完成度だけではない。

人間がプレイし、ズレや癖や選択が記録される。それをそのまま捨てず、次のゲーム体験に戻す。この流れが、AIエージェントとの協働、フィードバックの道具化、開発過程のポートフォリオ化という初期の物語に合っている。

ただし、人間のログをそのまま譜面にすると、音楽的に気持ち悪くなる可能性がある。そこで、ログをそのまま正本にせず、最小限の整形を通して「痕跡譜面」にする。

## 重要な割り切り

### やること

- 通常プレイ中のタップ時刻とレーンを保存する。
- プレイ終了時にローカルへJSON保存する。
- 直近ログから、次回用の簡易譜面を作る。
- メニューから通常譜面と痕跡譜面を選べるようにする。
- 痕跡譜面は、プレイヤーの前回の癖が少し見える程度でよい。

### やらないこと

- 全プレイヤーのログを集める。
- サーバーへ保存する。
- 本格的なAI推論をゲーム内で動かす。
- 完璧なオンセット検出を行う。
- 高度な譜面エディタを作る。
- ログ一覧UIを作り込む。

GitHub Pagesは静的ホスティングなので、サーバー保存はしない。保存先はローカル開発時の `user://`、またはブラウザ内の永続領域に限定する。

## 構造

```text
main.gd
  通常プレイと痕跡譜面プレイを動かす本体

menu.gd
  通常プレイ / 痕跡譜面プレイ を選ぶ入口

charts/
  base_chart.json
    最初に遊ぶ基本譜面

user://trace_run_latest.json
  直近プレイのタップログ

user://trace_chart_latest.json
  直近ログから作った痕跡譜面
```

公開用リポジトリには、ユーザー個別の `user://` データは含めない。

## 実装案

### 1. 通常プレイのタップログを保存する

通常プレイ中、タップした瞬間に以下を記録する。

```json
{
  "lane": 1,
  "time": 12.34
}
```

判定成功だけでなく、空振りも記録する。

理由は、上手く叩けた場所だけではなく、「人間がそこに意図を置いた」こと自体が痕跡だから。

プレイ終了時に、以下の形で保存する。

```json
{
  "song": "kyrgyz_techno_anthem",
  "bpm": 140.0,
  "offset": 0.08,
  "duration": 85.872,
  "taps": [
    { "lane": 1, "time": 12.34 }
  ]
}
```

### 2. ログから簡単な譜面を作る

タップログをそのまま譜面にしない。最小限だけ整える。

変換ルール:

- 曲の最初と最後に近すぎるタップは除外する。
- 近すぎる連打はまとめる。
- レーンはタップしたレーンを採用する。
- 時刻はタップ時刻そのものを使うか、近い拍へ軽く寄せる。
- 玉数が少なすぎる場合は、通常譜面を使う。

最初の実装では、拍寄せは弱めでよい。人の痕跡が見えることを優先する。

### 3. メニューに「痕跡譜面で遊ぶ」を追加する

メニューに以下を追加する。

- ゲームをプレイ
- 痕跡譜面で遊ぶ

痕跡譜面がまだ無い場合は、押した時に通常譜面へ案内するか、通常譜面で開始する。

### 4. ゲーム本体の譜面読み込みを分岐する

通常プレイ:

- 基本譜面を使う。
- そのプレイのタップログを保存する。
- 終了時に痕跡譜面を生成する。

痕跡譜面プレイ:

- 直近の痕跡譜面を使う。
- そのプレイもログとして保存する。
- 終了時に次の痕跡譜面を更新する。

## 説明文の核

提出時には、以下のように説明できる。

> このゲームは、固定譜面だけで終わりません。プレイヤーのタップの痕跡を記録し、それを次の譜面に戻します。ただしログをそのまま使うとゲームとして崩れるので、最小限の変換ルールを通して、遊べる形に整えています。人間のフィードバックを記録し、AI協働で道具化する、という開発思想を小さなゲーム内に入れました。

この説明なら、ゲーム内でAIがリアルタイム推論しているとは言わない。正確で、かつ思想が伝わる。

## リスクと対策

### リスク1: 痕跡譜面が気持ちよくない

対策:

- 最初は「完璧な譜面」ではなく「痕跡が戻る体験」を優先する。
- あまりに少ないログや極端なログなら通常譜面に戻す。

### リスク2: Web保存が環境で不安定

対策:

- 最初はGodotの `user://` 保存に寄せる。
- GitHub Pagesで保存が難しい場合は、痕跡譜面機能をローカル実演用として扱う。

### リスク3: 実装が大きくなりすぎる

対策:

- ログ一覧、譜面エディタ、複数履歴、共有機能は作らない。
- 直近1回だけ保存する。

### リスク4: 既存の公開計画が遅れる

対策:

- まず動く最小形だけ作る。
- 見た目や結果画面の強化は後回し。

## 完了条件

この作業は、以下で完了とする。

- 通常プレイ中のタップが記録される。
- プレイ終了後、直近ログが保存される。
- 直近ログから痕跡譜面が作られる。
- メニューから痕跡譜面で遊べる。
- 痕跡譜面が無い時もゲームが壊れない。
- 仕組みを議事録またはREADMEで説明できる。
